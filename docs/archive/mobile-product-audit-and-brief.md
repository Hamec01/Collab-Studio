## 1. Короткий вывод

CollabStudio уже не является пустым прототипом. В репозитории есть рабочая основа приватной студии:

- авторизация через сессию и Google OAuth;
- проекты, участники и роли;
- треки и версии текста;
- комментарии к строкам;
- чат и задачи;
- аудиоверсии, внешние ссылки, A/B loop и таймкод-аннотации;
- уведомления;
- PostgreSQL/Prisma и self-hosted deployment.

Но приложенные макеты изображают более широкий продукт: публичную музыкальную платформу с лентой, поиском пользователей, профилями, подписками, лайками, прослушиваниями и публичными публикациями. Эти сущности и API сейчас отсутствуют.

Поэтому правильный следующий шаг — не копировать макет целиком. Сначала нужно превратить существующую приватную студию в качественное mobile-first веб-приложение. Публичный discovery-слой следует проектировать отдельным этапом после появления backend-контрактов.

## 2. Что в текущем коде хорошо

- Backend и frontend уже разделены по API.
- Основные модели совместной работы существуют и связаны с ролями.
- Аудио не является декоративной заглушкой: работает через реальный `<audio>`, есть seek, скорость, громкость, A/B loop и аннотации.
- Сервер ограничивает доступ к проектам участниками.
- Есть адаптивные Tailwind-классы и простое мобильное переключение трёх рабочих зон.
- Production build проходит.

Это надо сохранить. Переписывание backend или редактора с нуля сейчас не оправдано.

## 3. Что не так в текущем приложении

### Архитектура frontend

- `App.tsx` управляет авторизацией, загрузкой данных, выбранным проектом и треком, модалками, уведомлениями, загрузками и мобильной навигацией одновременно.
- Крупные компоненты `AudioPlayer`, `ProjectList` и `LyricsEditor` уже слишком велики для безопасного редизайна одним изменением.
- Навигация хранится в локальном state. У экранов нет URL, deep link и нормальной истории Back.
- Нет общего app shell, route-level экранов и устойчивой модели навигации.
- Кнопка «Обсудить» ищет DOM-элемент `mobile_discussion_tab`, которого в коде нет. Этот переход фактически не работает.

### Mobile UX

- Текущий mobile layout — переключение трёх desktop-колонок, а не самостоятельный мобильный UX.
- Bottom navigation имеет только три пункта, не показывает активное состояние и не учитывает полноценные маршруты.
- Нет safe-area отступов для iPhone, системной работы с экранной клавиатурой и явного использования `100dvh`.
- У редактора нет autosave. Сохранение создаёт новую версию вручную.
- Плеер не persistent между мобильными экранами и не имеет collapsed/expanded состояния.
- Комментарии открываются как отдельная рабочая панель, а не как мобильный bottom sheet.
- Нет полноценных loading/skeleton/empty/offline состояний.
- Часть icon-only кнопок не имеет доступного имени; touch targets местами меньше желательных 44×44 px.

### Надёжность разработки

- На чистом `npm ci` команда `npm run lint` падает, пока отдельно не выполнен `prisma generate`. После генерации Prisma lint проходит.
- Автотестов и E2E-тестов в репозитории нет.
- Нет CI-проверки ключевых мобильных потоков.
- Имя npm-пакета всё ещё `react-example`.

### Ограничение модели комментариев

Комментарий привязан к `lineIndex`. При вставке или удалении строк существующие комментарии могут начать указывать не на тот текст. Для серьёзной совместной редакции нужна более устойчивая привязка: стабильный ID блока/строки либо anchor с цитатой и логикой перепривязки.

## 4. Что не так или спорно в макетах

### Дублирование навигации

Верхнее горизонтальное меню и нижняя навигация повторяют «Главную», «Проекты», «Редактор» и «Чат». Пользователь получает два конкурирующих способа перейти в одно место.

Решение:

- bottom navigation — единственная глобальная навигация;
- верхний ряд — контекстные быстрые действия, а не второй navbar;
- «Редактор» открывает последний трек только как shortcut, но не существует без выбранного проекта/трека.

### «Папка» не должна объединять свои и публичные проекты

«Мои проекты», «Проекты, которыми со мной поделились» и «Публичные проекты других пользователей» имеют разные права, задачи и mental model.

Решение:

- экран `Projects`: сегменты «Мои» и «Со мной»;
- отдельный будущий экран `Discover`: публичные работы и пользователи;
- не маскировать отсутствие публичного API мок-карточками в production flow.

### Главная перегружена

На маленьком экране одновременно предлагаются логотип, три действия в шапке, шесть быстрых разделов, публичная карусель, свои проекты, новости и пять нижних пунктов. Визуально эффектно, но при ежедневной работе это высокая когнитивная нагрузка.

Для первой версии главная должна отвечать на три вопроса:

1. Над чем я недавно работал?
2. Где меня ждёт действие?
3. Как быстро создать или открыть трек?

### Tablet — не большой телефон

Диапазон 768–1023 px не следует принудительно показывать как один узкий поток. Для планшета полезен master-detail:

- список проектов/треков слева;
- workspace справа;
- обсуждение открывается sheet или дополнительной панелью в landscape.

### Макет обещает функции, которых нет

Сейчас отсутствуют:

- публичность проекта и moderation;
- общий каталог проектов;
- поиск пользователей/треков/проектов;
- likes, plays, follows;
- публичные профили и статистика;
- персонализированная публичная лента;
- rich-text форматирование текста.

Эти элементы нельзя включать в acceptance criteria мобильного редизайна без отдельного backend-этапа.

### «Приложение под телефон» пока означает mobile web

Текущий продукт — Vite SPA, не iOS/Android-приложение и пока не PWA: нет manifest и service worker. На ближайшем этапе корректная цель — responsive mobile web. Установка на главный экран и offline shell могут быть отдельным PWA-этапом.

## 5. Рекомендуемая информационная архитектура

### Глобальная нижняя навигация

1. Главная
2. Проекты
3. `+` — контекстное create menu
4. Inbox — уведомления, приглашения, непрочитанные обсуждения
5. Профиль

Поиск находится в шапке. Публичный `Discover` появится после backend-этапа и может заменить или расширить поиск.

### Главная v1

- «Продолжить работу» — последние 3–5 треков;
- «Требует внимания» — непрочитанные комментарии, задачи и приглашения;
- «Недавняя активность» по доступным пользователю проектам;
- явные loading, empty и error состояния.

Публичной карусели в v1 нет.

### Проекты

- сегменты «Мои» и «Со мной»;
- поиск по доступным проектам;
- карточки с cover, названием, ролью, количеством треков и временем обновления;
- project detail со списком треков, участниками и действиями по правам;
- создание проекта и трека через sheets/dialogs.

### Track Workspace

Маршрут: `/projects/:projectId/tracks/:trackId/:tab?`

Вкладки:

- Текст
- Аудио
- Обсуждение
- Задачи
- Версии

Шапка показывает back, проект/трек, статус сохранения и overflow menu.

Плеер:

- один экземпляр на уровне workspace;
- collapsed и expanded режимы;
- продолжает играть при переходе между вкладками одного трека;
- не перекрывает поле ввода и экранную клавиатуру;
- сохраняет выбранную аудиоверсию, скорость и loop в пределах сессии трека.

Комментарии к строке:

- индикатор рядом со строкой;
- bottom sheet с контекстом строки;
- добавление и resolve по текущим правам;
- до изменения backend явно учитывать риск смещения `lineIndex`.

## 6. Порядок реализации

### Этап 2. Mobile Projects

- Перенести существующие проекты, треки, создание и приглашения.
- Добавить корректные loading/empty/error состояния.
- Не добавлять публичные проекты.

### Этап 3. Mobile Track Workspace

- Перенести текст, версии, комментарии, чат, задачи, аудиоверсии и аннотации.
- Сделать persistent collapsed/expanded player.
- Сделать line comments через bottom sheet.
- Сохранить role-based ограничения.

### Этап 4. Главная, Inbox и профиль

- Строить только из уже доступных пользователю данных.

### Этап 5. Public discovery

Отдельно спроектировать schema, privacy, API, moderation и UI для публикаций, поиска, подписок, likes и play counts.

## 7. Улучшенный промпт для следующего этапа разработки

Ниже — промпт для coding agent. Его следует запускать поэтапно, начиная только с Этапа 0 и Этапа 1.

---

Ты работаешь в существующем репозитории CollabStudio. Это React 19 + Vite + TypeScript frontend, Express backend, PostgreSQL/Prisma и session-cookie auth. Не переписывай приложение с нуля и не заменяй backend. Сначала изучи текущие компоненты, API, роли доступа и git diff.

Цель текущей итерации: создать надёжную mobile-first архитектуру и shell для существующей приватной collaborative studio. Это responsive web app, а не native iOS/Android. Публичную соцсеть на этом этапе не реализовывать.

### Обязательные продуктовые решения

- Существующий функционал проектов, треков, текста, lyric versions, audio versions, line comments, chat, tasks, annotations, notifications и permissions должен сохраниться.
- Не создавать production-моки для публичных проектов, likes, plays, follows, публичных профилей или глобального поиска: backend этих функций пока не поддерживает.
- Глобальная навигация на mobile: Главная, Проекты, `+`, Inbox, Профиль.
- Верхняя шапка содержит контекстный title, поиск/уведомления только там, где они реально работают, и avatar/menu.
- Не дублировать глобальную навигацию горизонтальным рядом из тех же пунктов.
- На tablet использовать master-detail layout, а не растянутую телефонную колонку.
- Desktop flow не должен потерять существующие возможности.

### Архитектурные требования

1. Перед визуальной переработкой добавь route-based navigation с deep links и browser Back:
   - `/`
   - `/projects`
   - `/projects/:projectId`
   - `/projects/:projectId/tracks/:trackId/:tab?`
   - `/inbox`
   - `/profile`
2. Не добавляй новые screen-state переключатели в монолитный `App.tsx`.
3. Вынеси session/data orchestration в небольшие hooks/providers или route-level containers.
4. Создай переиспользуемые primitives: app shell, screen header, bottom nav, tabs, sheet/dialog, loading state, empty state и error state.
5. Не дублируй server state в нескольких компонентах. Сохрани текущий API client как единственную границу backend.
6. Не меняй Prisma schema и API-контракты без явного обоснования и отдельного migration plan.

### Track Workspace

Сделай мобильное рабочее пространство трека с вкладками:

- Текст
- Аудио
- Обсуждение
- Задачи
- Версии

Один persistent audio player располагается на уровне workspace. Он имеет collapsed и expanded режимы, не размонтируется при смене вкладки текущего трека и не перекрывает экранную клавиатуру. Сохрани текущие возможности play/pause, seek, volume, speed, A/B loop, выбор аудиоверсии и таймкод-аннотации.

В режиме «Текст» сразу виден текущий текст. Редактирование использует существующую plain-text модель; не изображай rich text, пока его нет в данных. Сохрани ручное создание версии. Если добавляешь autosave draft, оно не должно создавать новую lyric version на каждый debounce и требует отдельной понятной семантики.

Комментарии к строке открывай в bottom sheet. Сохрани текущие permissions и resolve. Не скрывай риск текущей привязки по `lineIndex`; не меняй модель якорей в рамках UI-итерации.

### Responsive и accessibility

- Mobile baseline: 320–767 px.
- Tablet: 768–1023 px, предпочтительно master-detail.
- Desktop: 1024 px и выше.
- Используй `min-height: 100dvh`, `env(safe-area-inset-*)` и keyboard-safe layout.
- Интерактивные touch targets — не меньше 44×44 CSS px.
- Icon-only controls имеют `aria-label`.
- Навигация работает с клавиатуры и screen reader.
- Соблюдай `prefers-reduced-motion`.
- Не блокируй zoom.

### Состояния и ошибки

Для каждого route-level экрана реализуй:

- loading/skeleton;
- empty state с понятным следующим действием;
- recoverable error с retry;
- permission/read-only state;
- session-expired flow без потери понятного контекста.

### Защитная сетка до рефакторинга

- Исправь `npm run lint` на чистом `npm ci`: Prisma Client должен генерироваться до TypeScript typecheck.
- Добавь тесты маршрутов и критичных UI-state переходов.
- Добавь Playwright smoke tests минимум для:
  1. login;
  2. открытие проекта и трека по URL;
  3. редактирование и сохранение новой версии текста;
  4. открытие комментариев выбранной строки;
  5. запуск аудио и переход между вкладками без остановки;
  6. upload error;
  7. mobile bottom navigation;
  8. desktop regression.

### Порядок работы

1. Покажи краткий audit затрагиваемых файлов и план небольших изменений.
2. Реализуй только Этап 0 (тестовая/сборочная защитная сетка).
3. Проверь lint, build и tests.
4. Затем реализуй только Этап 1 (routes + responsive shell), не переделывая содержимое всех экранов.
5. Снова проверь lint, build, tests и Playwright на 390×844, 768×1024 и desktop.
6. Сообщи, какие функции пока используют старые компоненты и что остаётся на следующую итерацию.

### Definition of Done текущей итерации

- Clean install → lint → build → tests проходят в правильном порядке.
- У основных экранов есть URL и работает Back.
- Mobile shell не содержит дублирующей навигации.
- Bottom nav учитывает safe area и показывает активный route.
- Tablet имеет осмысленную адаптацию.
- Существующий desktop функционально не сломан.
- Нет фиктивных публичных данных и кнопок без поведения.
- Изменения разбиты на обозримые компоненты; `App.tsx` становится меньше, а не больше.
- В конце приложены результаты команд и список проверенных viewport.

## 8. Критерий хорошего результата

Хороший результат — не максимальное визуальное сходство с концепт-артом. Хороший результат — когда музыкант с телефона может за несколько секунд открыть нужный трек, читать и менять текст, слушать нужную аудиоверсию, оставить точный комментарий или таймкод и вернуться назад без потери контекста.
